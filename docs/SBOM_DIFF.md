# SBOM Diff Guide

OCIDex can diff any two SBOMs for the same artifact and present the result as either a dependency tree or a flat changelog. This guide explains how to read both views, how package matching works, and how to get the best results from your ingest pipeline.

## Tree View vs. Flat View

The diff page opens in **tree view** by default when the SBOM has a `dependencies` array with a root component (`metadata.component.bom-ref`). Tree view shows only the direct dependencies of the subject image at the top level; their changed descendants roll up via count chips.

**Flat view** shows every changed package as a flat list — no hierarchy. It is the fallback when no dependency graph is present, and it remains accessible at any time via the view-mode toggle. Your preference is saved in `localStorage`.

When the SBOM has a `dependencies` array but the tree would be empty (e.g., all changes are orphaned removed packages), the UI shows an explicit empty state rather than a blank tree, and offers a link to the flat list.

## Reading the Tree

Each row in tree view represents a node in the dependency graph.

| UI element | Meaning |
|---|---|
| Row indent level | Depth from root component (direct deps at depth 1) |
| Rollup chips (`↑3 ↓1 +2`) | Aggregate change counts across all transitive descendants of this node |
| Green badge | Package added |
| Red badge | Package removed |
| Blue badge | Package upgraded |
| Orange badge | Package downgraded |
| Gray badge | Package modified (version changed but direction unknown) |
| No badge | Package unchanged; visible when **Show context** is on |

### Controls

| Toggle | Default | Effect |
|---|---|---|
| Show transitive | Off | Include all transitive dependencies; off shows only direct deps and their changed subtrees |
| Show context | Off | Show unchanged nodes for structural context; on makes the full tree visible |
| Expand all changed | — | One-click button; expands every branch that has at least one changed descendant |

All control states are persisted in `localStorage` per artifact.

### Summary header

The header above the tree shows the total count of `added`, `removed`, `upgraded`, `downgraded`, and `modified` packages. These numbers reconcile with the rollup chips — the chips are computed by the same server-side DFS that produces the header totals.

## Component Identity Rules

The diff engine must decide whether a component in the "before" SBOM is the same package as a component in the "after" SBOM. Getting this wrong produces phantom remove+add pairs instead of upgrade events. The rules are applied on the backend; see [ADR-0019](adr/0019-diff-identity-model.md) for the full normative spec.

### Rule 1 — Purl primary key

When a component has a purl, its identity is the purl with:

- the `@version` segment removed, and
- only the qualifiers that are part of the package's provenance retained: `distro`, `arch`, `epoch`, `repository_url`.

Everything else (`download_url`, `checksum`, `tag`, `commit`, unknown qualifiers) is discarded — mirror changes and content-addressed identifiers must not break identity.

Retained qualifiers are sorted alphabetically so two purls differing only in qualifier order produce the same key.

**Practical impact on Syft output:**

- `pkg:apk/wolfi/curl@8.6.0?distro=wolfi-os` and `pkg:apk/alpine/curl@8.7.0?distro=alpine-3.19` have different identities — a Wolfi package will never match an Alpine package even if they share a name.
- `pkg:deb/ubuntu/curl@7.81.0-1?arch=amd64` and `pkg:deb/ubuntu/curl@7.81.0-2?arch=amd64` match — one `modified` event, not a remove+add.
- `pkg:apk/wolfi/curl?download_url=https://mirror-a.example` and `pkg:apk/wolfi/curl?download_url=https://mirror-b.example` match — `download_url` is noise.

### Rule 2 — Tuple fallback

When a component has no purl, identity falls back to `(type, group, name)`. This is a coarser key: two packages with the same name in different distros will match. Prefer SBOMs generated by tools that emit purls (Syft default, apko, recent Trivy).

### Rule 3 — Versioned-name post-pass

After the initial diff pass, a second pass collapses pairs where one package was removed and a different package was added, and the two names differ only in a trailing numeric version suffix (e.g., `go-1.24` and `go-1.25`, `python-3.11` and `python-3.12`).

These are reported as a single `upgraded` or `downgraded` event rather than separate `removed`/`added` entries.

**Survivor guard:** if multiple packages share the same normalized base name on either side (e.g., `gcc-11` and `gcc-12` are both installed), no collapse fires. This prevents `gcc-11` → `gcc-13` being incorrectly merged when `gcc-12` survived.

### What this means for your ingest pipeline

- **Always generate SBOMs with purls** (Syft does this by default). Without purls the diff engine falls back to tuple identity and cross-distro phantom matches are possible.
- **Ensure `?distro` qualifiers are present on apk purls.** Syft emits these automatically. Hand-written or Trivy-generated SBOMs may omit them, causing Alpine/Wolfi packages to look identical.
- **The versioned-name heuristic is a fallback, not the primary path.** If your toolchain produces `go-1.24` as the package name for Go's standard library, the collapse will work — but using purls (`pkg:golang/std@1.24`) is more precise.

## Flavor Axis

A single artifact can ship multiple variants with the same version tag but different base distros (e.g., `myapp:1.2.3-alpine` vs `myapp:1.2.3-debian`). Without a flavor axis, the changelog would interleave Alpine and Debian packages on the same timeline, making every package look added or removed on every release.

OCIDex detects the flavor of each SBOM at ingest time and stores it on `sbom.flavor`. Changelogs are grouped by `(architecture, flavor)` so each distro variant gets its own timeline. See [ADR-0020](adr/0020-image-flavor-axis.md) for the full normative spec.

### Flavor detection layers

Detection runs at ingest time from the SBOM body itself. Three layers are tried in order:

**Layer 1 — OS metadata properties (primary)**

Syft, Trivy, and apko embed OS info in CycloneDX properties:
- `syft:image:os:name` + `syft:image:os:version-id` → e.g., `alpine-3.19`, `debian-bookworm`
- `syft:distro:id` / `syft:distro:version-id` (older Syft)
- Trivy uses `aquasecurity:trivy:` prefixed equivalents

This is the most precise layer: it distinguishes `alpine-3.18` from `alpine-3.19`.

**Layer 2 — Purl-type fingerprint (secondary)**

When OS metadata is absent, the dominant purl ecosystem determines the flavor:

| Dominant purl type | Disambiguator | Detected flavor |
|---|---|---|
| `apk` | `?distro=wolfi*` on majority of purls | `wolfi` |
| `apk` | `?distro=chainguard*` | `chainguard` |
| `apk` | otherwise | `alpine` |
| `deb` | namespace contains `ubuntu` | `ubuntu` |
| `deb` | otherwise | `debian` |
| `rpm` | namespace contains `fedora` | `fedora` |
| `rpm` | namespace contains `rhel`/`redhat`/`centos` | `rhel` |
| No OS-pm purls (only `golang`, `npm`, `pypi`, …) | — | `distroless` |
| Empty component list | — | `scratch` |

**Layer 3 — Tag suffix heuristic (last resort)**

When layers 1 and 2 cannot resolve, the `subjectVersion` field is scanned for known suffixes: `-alpine`, `-wolfi`, `-chainguard`, `-debian`, `-bookworm`, `-ubuntu`, `-jammy`, `-distroless`, `-scratch`, and others. Longest match wins.

If no layer resolves, the flavor is stored as `unknown` (not null).

### Flavor tab strip

When an artifact has SBOMs with multiple distinct flavors, a tab strip appears at the top of the changelog page. Selecting a tab filters the view to that flavor's timeline. The currently selected flavor is shown in the image picker label so you always know which variant you're viewing.

### Recommendations

- **Tag images with a flavor suffix** (`myapp:1.2.3-alpine`, `myapp:1.2.3-debian`) so layer-3 detection covers cases where OS metadata is absent or stripped.
- **Use a recent Syft or apko** to generate SBOMs. These tools emit `syft:image:os:name`, `syft:image:os:version-id`, and `?distro` qualifiers — the two highest-confidence layers.
- **For multi-arch images**, each architecture generates a separate SBOM and a separate flavor detection run. Architecture and flavor are independent axes.
- **`unknown` flavor** is a deliberate sentinel meaning none of the three layers could determine the distro. The changelog will still work, but multi-flavor artifacts will be lumped together under `unknown` rather than separated by distro line.

## Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| "No dependency tree available — showing flat list" | SBOM has no `dependencies` array, or `metadata.component.bom-ref` is absent/not referenced as a root | Regenerate the SBOM with Syft ≥ 0.97 using `--output cyclonedx-json`; ensure `--source-name` / `--source-version` produce a `metadata.component` entry |
| Every package looks added/removed when comparing two releases of the same image | Multi-flavor images ingest into the same timeline | Check that Syft OS metadata is present (Layer 1); or add a flavor suffix to the image tag so Layer 3 can distinguish variants |
| A Go/Python/LLVM version bump shows as `removed X + added Y` instead of an upgrade | Versioned-name collapse was suppressed by the survivor guard | Both major versions were present simultaneously on one side; the two-slot case is intentionally kept as separate events (correct behavior) |
| `isDirect` shows nothing / "Show transitive" reveals no additional packages | `metadata.component.bom-ref` is absent or not referenced in the `dependencies` array root node | Use Syft ≥ 0.97; the `bom-ref` on `metadata.component` must appear as the root entry in `dependencies` |
| Alpine and Wolfi packages diff against each other (phantom remove+add across distros) | `?distro` qualifier missing from apk purls | Use Syft ≥ 0.90 or apko — both emit `?distro=wolfi-os` / `?distro=alpine-3.x` qualifiers on apk purls |
| Rollup chip totals don't match the flat-list count | None — they shouldn't differ | File a bug; the backend DFS and the flat list share the same change set |
